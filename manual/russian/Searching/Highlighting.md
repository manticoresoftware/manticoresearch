# Подсветка

<!-- example highlighting -->

Подсветка позволяет получить выделенные фрагменты текста (называемые сниппетами) из документов, содержащих совпадающие ключевые слова.

Функция SQL `HIGHLIGHT()`, свойство `"highlight"` в JSON-запросах через HTTP и функция `highlight()` в PHP-клиенте используют встроенное хранение документов для получения оригинального содержимого поля (включено по умолчанию).

<!-- intro -->
##### SQL:

<!-- request SQL -->

```sql
SELECT HIGHLIGHT() FROM books WHERE MATCH('try');
```

<!-- response SQL -->

```sql

+----------------------------------------------------------+

+----------------------------------------------------------+
1 row in set (0.00 sec)
```

При использовании SQL для подсветки результатов поиска вы получите сниппеты из различных полей, объединённые в одну строку из-за ограничений протокола MySQL. Вы можете настроить разделители объединения с помощью опций `field_separator` и `snippet_separator`, как описано ниже.

При выполнении JSON-запросов через HTTP или использовании PHP-клиента таких ограничений нет, и в результатах возвращается массив полей, содержащих массивы сниппетов (без разделителей).

Учтите, что опции генерации сниппетов, такие как `limit`, `limit_words` и `limit_snippets`, применяются по умолчанию к каждому полю отдельно. Вы можете изменить это поведение, используя опцию `limits_per_field`, но это может привести к нежелательным результатам. Например, в одном поле могут быть совпадающие ключевые слова, но сниппеты из этого поля не включаются в результат, так как они ранжируются ниже сниппетов из других полей в механизме подсветки.

Алгоритм подсветки в настоящее время отдаёт приоритет лучшим сниппетам (с более близким совпадением фразы), а затем сниппетам с ключевыми словами, которые ещё не включены в результат. В целом он старается выделить лучшее совпадение с запросом и выделить все ключевые слова запроса в пределах допустимых ограничений. Если в текущем поле совпадений не найдено, начало документа по умолчанию обрезается согласно ограничениям и возвращается. Чтобы вместо этого возвращать пустую строку, установите опцию `allow_empty` в 1.

Подсветка выполняется на стадии так называемого "post limit", что означает, что генерация сниппетов откладывается не только до подготовки полного итогового набора результатов, но и после применения оператора LIMIT. Например, с оператором LIMIT 20,10 функция `HIGHLIGHT()` будет вызвана не более 10 раз.

## Опции подсветки

<!-- example highlighting options -->

Существует несколько опциональных параметров подсветки, которые можно использовать для точной настройки генерации сниппетов и которые общие для SQL, HTTP и PHP клиентов.

#### before_match
Строка, вставляемая перед совпадением ключевого слова. В этой строке можно использовать макрос `%SNIPPET_ID%`. Первое появление макроса заменяется на увеличивающийся номер сниппета внутри текущего сниппета. Нумерация начинается с 1 по умолчанию, но может быть переопределена опцией `start_snippet_id`. `%SNIPPET_ID%` сбрасывается при начале каждого нового документа. Значение по умолчанию — `<b>`.

#### after_match
Строка, вставляемая после совпадения ключевого слова. Значение по умолчанию — `</b>`.

#### limit
Максимальный размер сниппета в символах (кодовых точках). Значение по умолчанию — 256. По умолчанию применяется к каждому полю отдельно, см. `limits_per_field`.

#### limit_words
Ограничивает максимальное количество слов, которые могут быть включены в результат. Обратите внимание, что это ограничение касается всех слов, а не только совпадающих ключевых слов для подсветки. Например, если подсвечивается слово `Mary` и выбран сниппет `Mary had a little lamb`, в это ограничение учитываются все 5 слов, а не только 1. Значение по умолчанию — 0 (ограничение отсутствует). По умолчанию применяется к каждому полю отдельно, см. `limits_per_field`.

#### limit_snippets
Ограничивает максимальное количество сниппетов, которые могут быть включены в результат. Значение по умолчанию — 0 (без ограничения). По умолчанию применяется к каждому полю отдельно, см. `limits_per_field`.

#### limits_per_field
Определяет, работают ли `limit`, `limit_words` и `limit_snippets` как отдельные ограничения для каждого поля документа, подсвечиваемого в отдельности, либо как глобальные ограничения для всего документа в целом. Установка этой опции в 0 означает, что все объединённые результаты подсветки для одного документа должны соответствовать указанным ограничениям. Минус в том, что в одном поле может быть выделено несколько сниппетов, а в другом — нет, если движок подсветки решит, что они более релевантны. Значение по умолчанию — 1 (использовать ограничения по каждому полю).
#### around
Количество слов, выбираемых вокруг каждого блока совпадающего ключевого слова. Значение по умолчанию — 5.

#### use_boundaries
Определяет, следует ли дополнительно разбивать сниппеты по границам фраз, как задано в настройках таблицы с помощью директивы [phrase_boundary](../Creating_a_table/NLP_and_tokenization/Low-level_tokenization.md#phrase_boundary). Значение по умолчанию — 0 (не использовать границы).

#### weight_order
Указывает, сортировать ли извлечённые сниппеты по релевантности (по убыванию веса) или по порядку появления в документе (по возрастанию позиции). Значение по умолчанию — 0 (не использовать порядок по весу).

#### force_all_words
Игнорирует ограничение длины до тех пор, пока результат не будет содержать все ключевые слова. Значение по умолчанию — 0 (не форсировать все ключевые слова).

#### start_snippet_id
Устанавливает начальное значение макроса `%SNIPPET_ID%` (который обнаруживается и разворачивается в строках `before_match` и `after_match`). Значение по умолчанию — 1.

#### html_strip_mode
Определяет режим удаления HTML-разметки. По умолчанию `index`, что означает использование настроек таблицы. Другие значения включают `none` и `strip`, которые принудительно пропускают или применяют удаление независимо от настроек таблицы; а также `retain`, который сохраняет HTML-разметку и защищает её от подсветки. Режим `retain` можно использовать только при подсветке полноценных документов, поэтому требуется отсутствие ограничений по размеру фрагментов. Допустимые строковые значения: `none`, `strip`, `index` и `retain`.

#### allow_empty
Позволяет возвращать пустую строку как результат подсветки, если в текущем поле не удалось создать фрагменты (нет совпадения по ключевым словам или фрагменты не помещаются в ограничение). По умолчанию возвращается начало исходного текста вместо пустой строки. Значение по умолчанию 0 (не разрешать пустой результат).

#### snippet_boundary
Гарантирует, что фрагменты не будут пересекать границы предложения, абзаца или зоны (при использовании с таблицей, у которой включены соответствующие настройки индексации). Допустимые значения: `sentence`, `paragraph` и `zone`.

#### emit_zones
Вставляет HTML-тег с именем окружающей зоны перед каждым фрагментом. Значение по умолчанию 0 (не вставлять имена зон).

#### force_snippets
Определяет, нужно ли принудительно создавать фрагменты, даже если лимиты позволяют подсветить весь текст. Значение по умолчанию 0 (не принуждать создание фрагментов).

В дополнение к общим параметрам подсветки, для JSON-запросов через HTTP доступны несколько синонимов:

#### fields
Объект `fields` содержит имена атрибутов с опциями. Также это может быть массив имен полей (без каких-либо опций).

Обратите внимание, что по умолчанию подсветка пытается выделить результаты согласно полнотекстовому запросу. В общем случае, если вы не указываете поля для подсветки, подсветка основывается на вашем полнотекстовом запросе. Однако если вы указываете поля для подсветки, подсветка происходит только если полнотекстовый запрос совпадает с выбранными полями.

#### encoder
Свойство `encoder` может быть установлено в `default` или `html`. При установке в `html` сохраняется HTML-разметка при подсветке. Это работает аналогично опции `html_strip_mode=retain`.

<!-- example highlight_query -->
#### highlight_query

## CALL SNIPPETS

<!-- example CALL SNIPPETS -->

Выражение `CALL SNIPPETS` формирует фрагмент из предоставленных данных и запроса, используя настройки указанной таблицы. Оно не может обращаться к встроенному хранилищу документов, поэтому рекомендуется использовать функцию [HIGHLIGHT()](../Searching/Highlighting.md).

Синтаксис:

```sql
CALL SNIPPETS(data, table, query[, opt_value AS opt_name[, ...]])
```

#### data
`data` служит источником, из которого извлекается фрагмент. Это может быть одна строка или список строк в фигурных скобках.
#### table
`table` — это имя таблицы, предоставляющей настройки обработки текста для генерации фрагментов.
#### query
`query` — это полнотекстовый запрос, используемый для построения сниппетов.
#### opt_value и opt_name
`opt_value` и `opt_name` представляют [опции генерации сниппетов](../Searching/Highlighting.md).

<!-- intro -->
##### SQL:
<!-- request SQL -->

```sql
CALL SNIPPETS(('this is my document text','this is my another text'), 'forum', 'is text', 5 AS around, 200 AS limit);
```

Большинство опций совпадают с [функцией HIGHLIGHT()](../Searching/Highlighting.md). Однако есть несколько опций, которые можно использовать только с `CALL SNIPPETS`.

<!-- example CALL SNIPPETS load files -->
Следующие опции можно использовать для подсветки текста, хранящегося в отдельных файлах:

#### load_files
Эта опция, при включении, рассматривает первый аргумент как имена файлов, а не данные для извлечения сниппетов. Указанные файлы на стороне сервера будут загружены для получения данных. Для параллелизации работы при включенном этом флаге будет использоваться до [max_threads_per_query](../Server_settings/Searchd.md#max_threads_per_query) рабочих потоков на запрос. По умолчанию 0 (ограничений нет). Для распределения генерации сниппетов между удалёнными агентами вызовите генерацию сниппетов в распределенной таблице, содержащей только одного(!) локального агента и нескольких удалённых. Опция [snippets_file_prefix](../Creating_a_table/Creating_a_distributed_table/Remote_tables.md#snippets_file_prefix) используется для генерации итогового имени файла. Например, когда searchd настроен с `snippets_file_prefix = /var/data_` и в качестве имени файла передаётся `text.txt`, сниппеты будут сгенерированы из содержимого файла `/var/data_text.txt`.

#### load_files_scattered
Эта опция работает только с распределённой генерацией сниппетов с удалёнными агентами. Исходные файлы для генерации сниппетов могут быть распределены по разным агентам, а главный сервер объединит все результаты без ошибок. Например, если у одного агента распределённой таблицы есть `file1.txt`, а у другого — `file2.txt`, и вы используете `CALL SNIPPETS` с обоими этими файлами, searchd объединит результаты агентов, так что вы получите результаты от обоих файлов `file1.txt` и `file2.txt`. По умолчанию 0.

Если опция `load_files` также включена, запрос вернет ошибку, если какой-либо из файлов недоступен. В противном случае (если `load_files` не включена) он вернет пустые строки для всех отсутствующих файлов. Searchd не передает этот флаг агентам, поэтому агенты не вызовут критическую ошибку, если файл не существует. Если вы хотите быть уверены, что все исходные файлы загружены, установите обе опции `load_files_scattered` и `load_files` в 1. Если отсутствие некоторых исходных файлов на некоторых агентах не критично, установите только `load_files_scattered` в 1.
